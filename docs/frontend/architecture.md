# RADAR Frontend — Architecture

`apps/web` is a stateless React + Vite single-page app. It **consumes the radar-api REST
routes** and never re-implements backend logic (evaluation is `@radar/engine`,
server-side). The NS1 key never reaches the browser — the SPA calls same-origin `/api`,
proxied to radar-api by Vite (dev) or nginx (prod).

## Composition
- **API client** (`src/api/`) — a thin typed `fetch` wrapper mirroring the API contracts.
- **Auth** (`src/auth/AuthContext`) — fetches `/api/v1/me`; exposes `hasPermission`.
  **RBAC in the UI is cosmetic** — the API enforces every permission server-side; the UI
  only hides controls a user cannot use.
- **Routing** — `react-router` with a shell layout (`AppShell`) whose navigation is
  permission-filtered.
- **Features** — Explain (`EvaluationView`), Steering, Topology.

## NS1 / Cloudflare responsibility boundary
This is the central modelling rule, made explicit in the Topology and Explain screens:

- **NS1 selects the delivery *platform*** — one of Réalta / Fastly / Akamai / CloudFront.
- **Cloudflare then selects the Réalta *origin pool*** (Donnybrook Pool 1/2, External Pool
  1/2 → Origin).
- **NS1 never selects a cache or pool.** The topology renders platform selection and pool
  selection as separate sections with a boundary note; the steering section contains no
  pool nodes.

## Configured vs observed
The delivery topology (`src/topology/model.ts`) is **RADAR-CONFIGURED / MANUALLY
MAINTAINED** architecture data — capacities, targets, and ASN→path mappings — **not live
measurement**. Every value is labelled `CONFIGURED` or `MANUALLY MAINTAINED`, and every
node/link shows *Utilisation: Telemetry not connected* plus its expected future source.
The UI never fabricates a utilisation percentage.

## Steering-matrix derivation
Steering rows are **generated by calling `/api/v1/dns/explain`** per scenario
(`src/steering/scenarios.ts`) — outcomes are never hard-coded. Each row shows the
API-derived identity source, eligible platforms, expected (probabilistic) distribution,
plus the configured network path/target and provenance.

## Partial-evaluation handling
When the API reports `complete: false` (an unsupported filter), the row/label shows
**Partial** and **no definitive platform or distribution** is asserted. Expected shares
are always labelled **probabilistic** (weighted shuffle), never a guaranteed traffic
split.

## Future telemetry sources (not yet connected)
Varnish telemetry (cache utilisation), router/interface telemetry (PNI/transit
utilisation), Cloudflare API (pool/edge), and a network telemetry/monitoring adapter.
These replace the *Telemetry not connected* placeholders in a later version.
